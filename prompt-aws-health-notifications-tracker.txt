## PROJECT OVERVIEW

Build an **AWS Health Notifications Tracker** that:
- Aggregates AWS Health events across ~200 AWS accounts in an AWS Organization
- Stores notifications with deadlines, recommended actions, and status tracking
- Provides a searchable, sortable React frontend served via CloudFront
- Uses LLM (Amazon Bedrock) to generate plain-English summaries and action items
- Sends reminders for approaching deadlines
- Tracks follow-up status per account/event

## ARCHITECTURE

### Data Flow
1. EventBridge Scheduler triggers `health-collector` Lambda every 15 minutes
2. `health-collector` calls AWS Health API (Organizations endpoint) to fetch all health events across all member accounts
3. New/updated events are written to DynamoDB; raw event JSON is archived to S3
4. DynamoDB Streams trigger `llm-summarizer` Lambda for new events
5. `llm-summarizer` calls Amazon Bedrock (Claude claude-haiku-4-5-20251001 вЂ” lowest cost) to produce:
   - Plain-English summary (2-3 sentences)
   - Bullet-point recommended actions
   - Urgency classification (Critical / High / Medium / Low) based on deadline proximity
   - Extracted deadline date (if any) in ISO 8601
6. Summarized results are written back to the same DynamoDB item
7. EventBridge Scheduler triggers `deadline-reminder` Lambda daily at 09:00 UTC
8. `deadline-reminder` scans for events with deadlines within 7 days and status != "Resolved" and publishes to SNS topic (org admin team email)
9. React SPA (S3 + CloudFront) calls API Gateway в†’ `api-handler` Lambda for all read/write operations
10. Cognito User Pool handles authentication for the frontend

### Services Used
| Service | Purpose |
|---|---|
| AWS Health API (Organizations) | Source of all health events across 200 accounts |
| Lambda (4 functions) | health-collector, llm-summarizer, deadline-reminder, api-handler |
| DynamoDB (1 table, on-demand) | Primary store for health events with status tracking |
| S3 (2 buckets) | Raw event archive + React SPA static assets |
| CloudFront | CDN for React SPA with OAC |
| API Gateway (HTTP API) | REST backend for the React app |
| Cognito User Pool | Frontend authentication |
| Bedrock (Claude Haiku) | LLM summarization and action extraction |
| EventBridge Scheduler | Cron triggers for health-collector and deadline-reminder |
| DynamoDB Streams | Trigger llm-summarizer on new events |
| SNS | Deadline reminder email notifications |
| CloudWatch Logs | All Lambda logs with 30-day retention |
| AWS CDK v2 | Infrastructure as Code |

## DYNAMODB TABLE DESIGN

**Table name**: `HealthEvents`
**Billing mode**: PAY_PER_REQUEST (on-demand)
**Primary key**: `eventArn` (String, partition key)
**Sort key**: `accountId` (String)

**Global Secondary Indexes**:
- `GSI1`: `service` (PK) + `startTime` (SK) вЂ” query by service sorted by date
- `GSI2`: `urgency` (PK) + `deadline` (SK) вЂ” query critical/high urgency items by deadline
- `GSI3`: `status` (PK) + `startTime` (SK) вЂ” query by open/resolved status

**Item attributes**:
```json
{
  "eventArn": "arn:aws:health:us-east-1::event/...",
  "accountId": "123456789012",
  "accountAlias": "my-prod-account",
  "service": "ECS",
  "eventTypeCode": "AWS_ECS_MAINTENANCE_SCHEDULED",
  "eventTypeCategory": "scheduledChange",
  "region": "us-east-1",
  "startTime": "2026-04-01T00:00:00Z",
  "endTime": "2026-05-01T00:00:00Z",
  "deadline": "2026-05-01T00:00:00Z",
  "lastUpdated": "2026-04-17T09:00:00Z",
  "statusCode": "open",
  "status": "Open",
  "affectedResources": ["arn:aws:ecs:us-east-1:123456789012:cluster/my-cluster"],
  "rawEventUrl": "s3://bucket/raw/eventArn.json",
  "awsHealthUrl": "https://health.aws.amazon.com/health/home#/account/dashboard/open-issues",
  "summary": "AWS is scheduling maintenance on your ECS clusters...",
  "recommendedActions": ["Action 1", "Action 2"],
  "urgency": "High",
  "followUpStatus": "Pending",
  "followUpNotes": "",
  "followUpOwner": "",
  "llmProcessed": true,
  "ttl": 1777836000
}
```

**TTL**: Set to 1 year from event `endTime` to auto-expire old resolved events.

## LAMBDA SPECIFICATIONS

### 1. `health-collector` Lambda

**Trigger**: EventBridge Scheduler вЂ” rate(15 minutes)
**Runtime**: Python 3.12
**Timeout**: 5 minutes
**Memory**: 512 MB
**IAM permissions**:
- `health:DescribeEventsForOrganization`
- `health:DescribeEventDetailsForOrganization`
- `health:DescribeAffectedAccountsForOrganizationEvent`
- `health:DescribeAffectedEntitiesForOrganization`
- `organizations:ListAccounts`
- `dynamodb:PutItem`, `dynamodb:UpdateItem`, `dynamodb:GetItem`
- `s3:PutObject`

**Logic**:
- Call `health.describe_events_for_organization()` with filter: `lastUpdatedTime` = last 15 minutes (use a DynamoDB item `cursor/last_run_time` as checkpoint)
- For each event, call `describe_event_details_for_organization()` to get full description
- Call `describe_affected_accounts_for_organization_event()` to get all affected account IDs
- For each account+event combination, upsert DynamoDB item
- Archive raw JSON to S3: `s3://archive-bucket/raw/{year}/{month}/{day}/{eventArn_accountId}.json`
- If no events in last 15 min, do a weekly full backfill scan (check DynamoDB for `cursor/last_full_backfill`, if >7 days old, run full scan of last 90 days)
- Update `cursor/last_run_time` in DynamoDB after successful run

**Error handling**: On failure, write error to CloudWatch and raise so EventBridge retries (max 3 retries with exponential backoff)

### 2. `llm-summarizer` Lambda

**Trigger**: DynamoDB Streams (TRIM_HORIZON, batch size 10, filter: `eventName` IN [INSERT, MODIFY] AND `newImage.llmProcessed` = false)
**Runtime**: Python 3.12
**Timeout**: 2 minutes
**Memory**: 256 MB
**IAM permissions**:
- `bedrock:InvokeModel` for `anthropic.claude-haiku-4-5-20251001`
- `dynamodb:UpdateItem`

**Logic**:
- For each record in the stream batch, extract the event's `eventTypeCode`, `service`, raw description text, `startTime`, `endTime`
- Build a prompt for Bedrock (Claude Haiku):
  ```
  You are an AWS cloud operations assistant. Given this AWS Health event, provide:
  1. A 2-3 sentence plain-English summary for a non-technical stakeholder
  2. A bulleted list of 3-5 specific recommended actions with clear steps
  3. Urgency classification: Critical (deadline <3 days), High (deadline 4-14 days), Medium (deadline 15-30 days), Low (deadline >30 days or no deadline)
  4. Extract the action deadline date in ISO 8601 format (or null if none)

  Respond ONLY as valid JSON: {"summary": "...", "recommendedActions": ["...", "..."], "urgency": "High", "deadline": "2026-05-01T00:00:00Z"}

  AWS Health Event:
  Service: {service}
  Event Type: {eventTypeCode}
  Start: {startTime}
  End: {endTime}
  Description: {description}
  ```
- Parse JSON response; handle malformed output with retry (max 2 retries)
- Update DynamoDB item: set `summary`, `recommendedActions`, `urgency`, `deadline`, `llmProcessed=true`

### 3. `deadline-reminder` Lambda

**Trigger**: EventBridge Scheduler вЂ” cron(0 9 * * ? *) (09:00 UTC daily)
**Runtime**: Python 3.12
**Timeout**: 1 minute
**Memory**: 128 MB
**IAM permissions**:
- `dynamodb:Query` (GSI2 on urgency + deadline)
- `sns:Publish`

**Logic**:
- Query DynamoDB GSI2 for all items where `urgency` IN [Critical, High] AND `deadline` between now and now+7days AND `followUpStatus` != "Resolved"
- Also query for items where `urgency` = "Medium" AND `deadline` between now and now+3days
- Group by account; build a digest email per account with event summaries and action links
- Publish to SNS topic `HealthDeadlineAlerts` with structured JSON payload
- Log count of reminders sent to CloudWatch metric `HealthTracker/RemindersPublished`

### 4. `api-handler` Lambda

**Trigger**: API Gateway HTTP API (Cognito JWT authorizer)
**Runtime**: Python 3.12
**Timeout**: 30 seconds
**Memory**: 256 MB
**IAM permissions**:
- `dynamodb:Query`, `dynamodb:GetItem`, `dynamodb:UpdateItem`, `dynamodb:Scan`

**Routes** (all require Cognito JWT):
```
GET  /events                    - List events with pagination (limit, lastKey), filter by: service, urgency, status, accountId, dateFrom, dateTo, search (full-text on summary)
GET  /events/{eventArn}/{accountId} - Get single event detail
PATCH /events/{eventArn}/{accountId} - Update followUpStatus, followUpNotes, followUpOwner
GET  /stats                     - Return counts: total, by urgency, by status, by service (top 10), upcoming deadlines count
GET  /accounts                  - Return list of distinct accountIds+aliases from DynamoDB
```

**Pagination**: Use DynamoDB `LastEvaluatedKey` encoded as base64 for cursor-based pagination.
**Search**: Implement case-insensitive substring match on `summary`, `eventTypeCode`, `service` using DynamoDB FilterExpression (not a separate search service вЂ” keep costs low).
**CORS**: Allow origin matching CloudFront distribution domain.
